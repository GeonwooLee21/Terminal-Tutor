# Terminal Keypad

> 터미널 명령어를 몰라도 버튼 클릭만으로 터미널을 조작할 수 있게 해주는 CLI 보조 도구

---

## 프로젝트 소개

우리는 컴퓨터를 실행하면 가장 먼저 바탕화면을 마주하고, 마우스 클릭으로 원하는 것을 조작한다. 사실 이 GUI 환경의 전신이 바로 터미널, 즉 CLI 환경이다. 그러나 GUI에 익숙해진 현대인에게 터미널은 낯설고 어려운 도구로 느껴진다.

하지만 컴퓨터를 사용하다 보면 예기치 못하게 터미널을 켜야 하는 순간이 찾아온다. 실제로 컴퓨팅사고와 SW코딩 중간고사 실습 평가 때 학생들은 Jupyter Notebook을 터미널로 실행해야만 했다. 이때 꽤 많은 학생들이 터미널 조작에 어려움을 겪었는데, 이러한 모습이 해당 프로젝트의 강력한 동기로 작용했다.

기존의 TUI 파일 매니저들은 Vim 키바인딩 기반이라 오히려 초보자에게 더 낯선 경험을 제공한다. 일반 사용자에게 터미널은 리모컨 하나 없이 마주하는 하얀/검은 화면에 불과하다. **Terminal Keypad는 OS가 컴퓨터 내부 구조를 모르는 사용자도 컴퓨터를 쉽게 사용할 수 있도록 돕듯, 리모컨처럼 버튼을 선택하는 것만으로 터미널을 조작할 수 있는 인터페이스를 제공해 터미널 비기너와 일반 사용자 모두의 진입장벽을 낮추는 것을 목표로 한다.**

---

## 핵심 기능

- **키패드 기반 명령어 실행** — `ls`, `cd`, `mkdir` 등 주요 명령어를 번호 입력만으로 선택하고 실행한다.
- **현재 위치 시각화** — 사용자가 터미널에서 현재 어느 디렉토리에 위치하는지를 경로 형태로 직관적으로 표시한다.
- **위험 명령어 실행 전 경고** — `rm` 등 복구가 어려운 명령어 실행 시, 사전 경고 메시지를 띄워 초보자의 실수를 방지한다.

터미널을 처음 접하는 비전공 학부생, 또는 Jupyter Notebook 실행처럼 특정 상황에서 터미널을 어쩔 수 없이 사용해야 하는 일반 사용자

## 실행 방법

**요구 사항**
- Python 3.10 이상
- macOS / Linux (터미널 환경)

**1. 저장소 클론**

```bash
git clone https://github.com/<your-username>/terminal-keypad.git
cd terminal-keypad
```

**2. 라이브러리 설치**

```bash
pip install rich
```

**3. 실행**

```bash
python main.py
```

**4. 조작 방법**

| 입력 | 동작 |
|------|------|
| 버튼 번호 (예: `1`) | 해당 명령어 실행 |
| `b` | 인자 선택 화면에서 키패드로 돌아가기 |
| `q` | 프로그램 종료 |

## 사용된 Python 개념

| 개념 | 활용 |
|------|------|
| 딕셔너리, 리스트, 함수 | 명령어 버튼 구조 및 키패드 로직 설계 |
| `subprocess` 모듈 | Python 코드 안에서 외부 프로세스(터미널 명령어 등)를 실행하고 그 결과를 받아온다. 사용자가 선택한 명령어를 실제 터미널에 실행하는 데 활용한다. |
| `rich` 라이브러리 | 터미널 안에 텍스트 기반 UI(버튼, 색상, 레이아웃 등)를 렌더링할 수 있게 해주는 라이브러리다. 키패드 화면 구성 및 위험 명령어 빨간색 강조에 활용한다. |
| 클래스 (OOP) | `Button` 클래스로 명령어의 이름·설명·위험 여부 등 속성과 `execute()` 메서드를 하나로 묶어 관리한다. |
| `os` 모듈 | 운영체제와 상호작용하는 기능을 제공한다. 현재 디렉토리 경로 탐색 및 파일·폴더 목록 조회에 활용한다. |
| 조건문 | 위험 명령어(`rm` 등) 실행 여부를 감지하고, 해당 명령어가 선택됐을 때 경고 메시지를 출력해 사용자의 최종 확인을 받는 데 활용한다. |

**1. 라이브러리 선택**
- **어디에 사용**: 프로젝트 구현에 필요한 라이브러리 선정
- **받은 도움**: 터미널 명령어 실행에 `subprocess`, 디렉토리 탐색에 `os`, 터미널 UI 렌더링에 `rich`가 적절하다는 판단을 제안받음
- **내 판단**: 각 라이브러리의 역할을 이해한 뒤 채택

---

**2. 라이브러리 사용법 학습**
- **어디에 사용**: `subprocess.run()`, `os.chdir()` / `os.listdir()` / `os.path.join()`, `rich`의 `Console` / `Table` 사용법
- **받은 도움**: 각 함수의 매개변수와 반환값, 사용 예시를 학습함
- **내 판단**: 공식 문서와 대조하며 확인 후 적용

---

**3. `get_arg_buttons()` 리스트 컴프리헨션 리팩토링**
- **어디에 사용**: `get_arg_buttons()` 함수 코드 개선
- **받은 도움**: for문 + append로 작성한 폴더 필터링 로직과 버튼 생성 로직을 더 간결하게 표현할 수 있는 리스트 컴프리헨션 방식을 제안받음
- **내 판단**: 두 곳의 리스트 컴프리헨션 문법을 검토 후 이해하고 채택. 조건 필터링이 포함된 `[e for e in entries if ...]` 형태와 객체 생성을 담은 `[Button(...) for t in targets]` 형태가 각각 어떻게 동작하는지 확인함

---

**4. `show_keypad()` 리스트 언패킹 (`*row`)**
- **어디에 사용**: `show_keypad()`에서 `table.add_row()`에 리스트를 넘기는 방법
- **받은 도움**: 리스트를 `*`로 언패킹해서 함수 인자로 전달하는 방법 학습
- **내 판단**: 동작 확인 후 채택. `*row`가 `row[0], row[1], row[2]`를 풀어서 넘기는 것과 동일하다는 걸 이해함

---

**5. `ARG_PROMPTS` 딕셔너리 설계**
- **어디에 사용**: `mkdir` / `touch` 분기 처리 설계
- **받은 도움**: 명령어별 안내 문구를 분기로 나누어 처리하지 말고, `ARG_PROMPTS` 딕셔너리를 이용해 분기를 나누지 않는 방식을 제안받음
- **내 판단**: 각각 `elif`로 분기하는 것보다 코드 중복이 줄어들고, 명령어가 늘어나도 딕셔너리 항목만 추가하면 된다는 걸 이해하고 채택. 동작 확인함

## 한계와 다음 단계

**현재 한계**
- macOS / Linux 전용이다. Windows 환경에서는 동작하지 않는다 (`ls`, `rm` 등 Unix 계열 명령어 사용).
- 키보드 번호 입력 방식이라, 실제 "버튼 클릭" GUI와는 차이가 있다.
- 지원하는 명령어가 `ls`, `pwd`, `cd`, `mkdir`, `touch`, `cat`, `clear`, `rm`, `rm -rf`로 제한되어 있다.
- 명령어 실행 결과가 긴 경우 터미널 화면이 많이 흐트러질 수 있다.

**다음 단계 (NICE / LATER 기능)**
- 사용자가 자주 쓰는 명령어를 즐겨찾기로 저장하는 기능
- 사용자가 원하는 명령어를 키패드에 직접 추가하는 기능 (사용자 정의 키패드)
- 터미널 내 실시간 키패드 GUI 구현 (버튼으로 명령어•옵션•인자를 조합하고 Enter로 실행, curses 기반)
- Windows 지원 (`dir`, `del` 등 Windows 명령어 자동 전환)
